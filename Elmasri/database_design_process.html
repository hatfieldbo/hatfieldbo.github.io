Database Design can be divided into 5 stages:

  1. Requirements Collection and Analysis
  2. Conceptual Design
  3. Logical Design
  4. Application and Security Design
  5. Physical Database Design

Requirements Collection and Analysis

  • Determine end-user views, outputs, and transaction requirements. What  do the users want from the database?
    • what types of data
    • what application
    • what operations and how they affect the performance

Conceptual Database Design

  • High-level description of the data to be stored

    • what constraints
    • methodologies
      • goal: to create a simple description of the data that closely match how users and developers think of the data
      • use high-level data models/semantic data models
      • e.g., In case of Relational modeling,
        • Define entities, attributes, domains and relationships
        • Draw ER/EER diagrams, normalize entity attributes

Logical Database Design (Database Schema)

  • Choose a DBMS to implement the conceptual design
  • Convert the conceptual database design into a schema (logical schema) in the DBMS chosen
    • Map conceptual model to logical model components
      • Define tables, attributes, relationships and constraits
    • Validate logical model using normalization
      • Normalized set of tables
    • Validate logical modeling integrity constraints
      • Ensure entity and referential integrity
      • Define column constraints
    • Validate logical model against user requirements
      • Ensure the model supports user requirements
  • Example: If relational data model is chosen, we need to convert ER diagram into a relational database schema
  • schema refinement: Analyze the logical schema obtained in order to

    • identify potential problems (e.g., performance criteria, security issues)

    • refine the logical schema

    • can be guided by theory (e.g., normalization theory)

Application and Security Design

  • Application-related (beyond the database itself)

  • Code each task in an application language (e.g., Java)

  • e.g., Enforcing access rules on entities:

    • which parts of the database should be accessible to the role of an entity

    • which parts should not be accessible to the role of an entity

Physical Database Design

  • Define data storage organization
    • Define tables, indexes, and view's physical organization
  • Design integrity and security measures
    • Define users, security groups, rolesm, and access controls
  • Determine performance measures
    • Define database and query execution parameters
  • e.g.,

    • building indexes on some tables

    • clustering some tables

    • redesign part of the schema (if needed)
